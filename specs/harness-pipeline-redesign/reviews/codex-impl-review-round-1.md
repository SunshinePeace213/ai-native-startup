## Blocking findings

1. **The required internal review cannot run under the draft-PR, model, and comment-ownership protocol.** `.claude/commands/build.md:40,59` and `.claude/commands/harness-layer/harness-build.md:42,61` open a draft PR and then invoke the installed official `/code-review`. That command's eligibility gate (`~/.claude/plugins/cache/claude-plugins-official/code-review/unknown/commands/code-review.md:11`) stops on draft PRs; it is a direct prompt with no Sonnet-lead enforcement and posts its own unmarked comment (`:14,28`). This can tick Internal code-review without a review and contradicts DESIGN CONTRACT points 4, 5, and 7. Fix: invoke a read-only Sonnet review lead with Haiku finders that reviews the supplied range, returns findings without `gh`, and lets the build lead upsert the sole `<!-- report:code-review -->` comment after recording the reviewed SHA.

2. **Existing marker comments cannot be updated with the prescribed CLI flag.** `.claude/commands/build.md:31`, `.claude/commands/harness-layer/harness-build.md:33`, `.claude/commands/plan-w-team.md:111,119`, and `.claude/commands/harness-layer/harness-plan.md:123,131` require locating a marker, PATCHing via `gh api`, and uploading every body with `--body-file`. The official [`gh api` manual](https://cli.github.com/manual/gh_api) has no `--body-file`; it supports file-backed `-F body=@<file>` or `--input`. The update half therefore cannot run literally, violating DESIGN CONTRACT points 3 and 5. Fix: specify creation with `gh issue/pr comment --body-file`, update with `gh api --method PATCH -F body=@<file>` (or a JSON `--input` file), and paginate marker discovery before choosing create versus update.

3. **Accepted post-approval advice can move the head without renewed approval.** `.claude/commands/build.md:70-72` and `.claude/commands/harness-layer/harness-build.md:72-74` record the approved/Ready SHA and ask about advisory recommendations, but define no accepted-advice transition to a new commit+push and another review. The plan commands do request another round, yet their two-round caps (`.claude/commands/plan-w-team.md:30,115-125`; `.claude/commands/harness-layer/harness-plan.md:32,127-137`) leave no valid R3 path after an R2 approval. This violates DESIGN CONTRACT point 2. Fix: process advice before Ready; when accepted, invalidate the prior approved head, make a separate `Refs #N` commit+push, and require a fresh N+1 delta round even beyond the ordinary blocker-round cap before recording the new report commit as approved.

Fix: make a justified prior-PASS/unchanged carry-forward an allowed recorded skip; only an affected, previously failing, or otherwise required command omitted this round is blocking.
