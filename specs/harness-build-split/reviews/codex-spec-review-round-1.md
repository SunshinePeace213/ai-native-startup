### Round 1 — Verdict: changes-requested

- **The hand-off SHA cannot be the final pushed SHA as specified.** spec.md requires build to write `Last pushed SHA: <sha>` into `## Tracking`, commit, and push that edit, but a commit cannot contain its own final SHA; tasks.md then sequences dev-log and memory writes after that push, so the recorded value is stale even if it names the pre-tracking commit. Fix: define an immutable pre-metadata `Build hand-off SHA` (or derive the PR head instead), move every build-side repository write before the terminal hand-off push, and align spec.md § Objective / Requirements, tasks.md Task 1, and AC4 with that definition.
- **The approved flow advances the PR after the approved head.** tasks.md Task 1 commits the approved report, verifies that commit as the approved head, and runs `gh pr ready`, but then renders and commits `dev-notes.html` on any verdict and runs a memory step that may also edit tracked files. This violates the objective that readiness occurs at the approved head and leaves post-review content on a ready PR. Fix: render/link dev notes and perform all tracked logging/memory edits before the terminal approval commit; then commit the report and terminal artifacts together, push, verify the remote PR head, and call `gh pr ready`, with no later repository mutations. Update the review-loop/dev-log requirements, Task 1, AC5, and AC8 accordingly.
- **The team orchestration cannot deploy the declared builders as written.** tasks.md says the lead never edits directly and uses `Task*` tools to deploy team members, then instructs it only to call `TaskCreate`; the cached official docs say the task list coordinates work after teammates exist, while teammates are spawned separately and an agent team forms when the first teammate is spawned (`ai-docs/anthropic/agent-teams.md`, “Assign and claim tasks” and “How Claude starts agent teams”). With no spawn step, the board has tasks but no workers and the build gets stuck. Fix: add explicit Agent-tool spawn steps for the named builders before assignment (and state the agent-teams prerequisite), or rewrite the orchestration for ordinary subagents without shared-team claims.
- **The validation commands do not prove several criteria they claim to verify.** For example, the AC3 command checks only the tidy marker and absence of `hold`, not tidy-before-PR ordering or simplifier/behavior-preserving instructions; the AC5 command checks only `sonnet` and `gh pr ready`, not retry, digest relay, two-round cap, failure hand-off, or approved-head equality; and the AC6 command does not check self-run validation or unexecutable-command blocking. Fix: expand acceptance-criteria.md § Validation Commands with deterministic ordering, exact-count, and required-contract checks for every AC clause, or narrow each command's mapping and add explicit observable checks for the uncovered clauses.

**Recommendations (advisory, non-blocking):**

- Add `ai-docs/anthropic/agent-teams.md` to `decisions.md` § KB References after reconciling the deployment contract; agent-team spawning is load-bearing but currently unreferenced.
- The exact `codex exec -C ... -s workspace-write` invocation is not grounded by the cached Codex files listed for this plan. Add the official Codex CLI reference through `/kb add` and verify those flags before build.

**Issue-comment digest:** Round 1, changes-requested — 4 blocking: the hand-off SHA is self-invalidating, post-approval mutations move the PR past its approved head, Task-only orchestration never spawns teammates, and the validation commands do not prove their claimed criteria. Next: revise those contracts and checks, then re-review.
