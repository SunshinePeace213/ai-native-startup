# Acceptance Criteria: c-suite CPO department

> The definition of "done" for [spec.md](./spec.md). Every criterion is observable and testable, and
> every task in tasks.md should map to at least one criterion here.

## Acceptance Criteria

- **AC1** — The three stage commands exist at `.claude/commands/c-suite/cpo-intake.md`, `cpo-prd.md`, `cpo-brief.md`; each has frontmatter with `description:` and `argument-hint:`, carries the house sections (a `#` Purpose heading, `## Variables`, `## Instructions`, `## Workflow`), and names at least one `cpo-*` knowledge skill it loads.
- **AC2** — The commands resolve under ONE canonical prefix proven on the machine-readable discovery surface — the `slash_commands` field of the headless init event, never a model reply: exactly three entries match `(c-suite:)?cpo-(intake|prd|brief)`, all under the same prefix form, and that form equals the `Canonical command prefix:` line recorded in `.claude/rules/c-suite/cpo-operations.md`. The three knowledge skills never auto-fire (frontmatter blocks model auto-invocation; the recorded legacy fallback applied and noted if needed).
- **AC3** — `cpo-intake.md` defines both modes (LIVE grilling; ASYNC question-list generation with send-to-client instructions) and carries explicit `Resume existing engagement`, `Ingest returned answers: discovery/client-answers.md`, `Late-answer transition: prd -> stale`, and `Late-answer transition: brief -> stale` workflow markers in that order, so re-entrancy and downstream invalidation are deterministic rather than inferred from keywords.
- **AC4** — `cpo-prd.md` carries exact `Automatic rounds: 1, 2`, `Verdict source: report file only`, `Silence: not approval`, `Round 2 options: final-delta-round | accept-with-noted-gaps | needs-human`, and `Round 3 options: accept-with-noted-gaps | needs-human` control-flow markers. In real mode it upserts each digest as the idempotent engagement-issue comment keyed `<!-- prd-review-round-N -->`, applies the needs-human label only for that exit, commits with `Refs #N`, pushes the engagement branch, and opens no PR; in fixture mode it retains the report and records the gate outcome in `status.md`, commits on the hosting branch with a recorded SHA, and performs no issue comment, label, push, or PR action.
- **AC5** — All nine engagement templates exist inside their owning skills: `cpo-question-bank/templates/{question-list,requirements,status}.md`, `cpo-prd-standard/templates/prd.md`, `cpo-design-standard/templates/{project-brief,sitemap-and-flows,section-briefs,copy-deck,brand-inputs}.md`; the status template carries exact-format per-stage state lines (`<stage>: <state>`), per-stage run logs `<stage>-run: date=<date>; sha=<sha>; lessons=<result>`, and exact `mode:`, `issue:`, `branch:`, `hosting-branch:`, and `PR:` fields. Real mode fills issue/engagement branch/PR and uses `hosting-branch: none`; fixture mode uses `issue: none`, `branch: none`, `PR: none`, and fills the hosting branch plus each run's SHA.
- **AC6** — The three rules files exist under `.claude/rules/c-suite/` (`roster.md`, `cpo-operations.md`, `cpo-lessons.md`), each with `paths:` frontmatter (path-scoped, not always-loaded); `roster.md` contains the exact seat/persona/type/model mappings CPO/Sofia Reyes/AI/`session`, PM/Ethan Park/AI/`opus`, UX Researcher/Priya Nair/AI/`sonnet`, UX Designer/Jonas Weber/AI/`sonnet`, UI Designer/Yuki Tanaka/AI/`sonnet`, Design Lead/Daniel Osei/AI/`opus`, and Prototype Owner/Human/human/`n/a`.
- **AC7** — The three knowledge skills pass the meta-skills validator and their frontmatter blocks auto-invocation.
- **AC8** — The five persona agents under `.claude/agents/cpo/` pass the meta-agent validator; none lists `AskUserQuestion` in `tools`; their exact `skills:` mappings are cpo-pm → cpo-prd-standard + cpo-design-standard, cpo-ux-researcher → cpo-question-bank, and cpo-ux-designer/cpo-ui-designer/cpo-design-lead → cpo-design-standard; their `model` aliases match the roster per agent (cpo-pm `opus`, cpo-design-lead `opus`, cpo-ux-researcher `sonnet`, cpo-ux-designer `sonnet`, cpo-ui-designer `sonnet`). Each stage command's delegation briefs carry its exact owning `${CLAUDE_PROJECT_DIR}/.claude/skills/<skill>/SKILL.md` path plus every template path it delegates against (intake: question-list/requirements/status; PRD: prd; brief: project-brief/sitemap-and-flows/section-briefs/copy-deck/brand-inputs); the smoke check covered whichever invocation-control field shipped — including, under the legacy fallback, a spot-check that a deployed persona Reads the brief-carried `SKILL.md`.
- **AC9** — `.agents/skills/prd-review/SKILL.md` mirrors the spec-review contract: first-line verdict `### Round N — Verdict: approved|changes-requested` (em-dash), single report file `products/<client-slug>/prd/reviews/codex-prd-review-round-N.md`, terse two-line CLI return, closing `**Issue-comment digest:**`, judges against `requirements.md` + the cpo-prd-standard structure, and never calls `gh`.
- **AC10** — `AGENTS.md` documents `products/` in Project Structure and points to the `.claude/rules/c-suite/` family and the CPO pipeline.
- **AC11** — The dry-run fixture `products/_example-bluebird-bakery/` is complete: `discovery/requirements.md` carries one exact `- <dimension>: locked` marker for each named dimension (business & goals, audience & users, brand & voice, content & assets, structure & pages, features & functionality, integrations & data, technical & hosting constraints, budget & timeline, success metrics, references & competitors, legal/compliance), `prd.md` plus an `approved` verdict report under `prd/reviews/`, all six design files present by name (`project-brief.md`, `sitemap-and-flows.md`, `section-briefs.md`, `copy-deck.md`, `brand-inputs.md`, ≥1 page under `design/wireframes/`), `status.md` carries the exact state lines `intake: done`, `prd: done`, `brief: done` and `engagement: handed-off`, and no wireframe references an external URL.
- **AC12** — `cpo-intake.md` states the canonical slug rule `^[a-z0-9][a-z0-9-]{0,38}[a-z0-9]$` with the `_example-` fixture exception and validates it at preflight BEFORE any filesystem/branch/worktree/`gh` side effect; the validation script's negative test invokes intake headlessly with a malformed slug and asserts a STOP with zero `products/` side effects.
- **AC13** — The fixture's `status.md` proves the fixture-mode trail: exact `mode: fixture`, `issue: none`, `branch: none`, and `PR: none` lines; a non-empty `hosting-branch:` naming a local branch; and non-empty `intake-run:`, `prd-run:`, and `brief-run:` entries each containing a SHA reachable from that recorded branch and a non-empty lessons-check result ("none" allowed). The three command files also carry their exact real/fixture trail-contract lines, proving real mode uses issue comments/labels where specified, `Refs #N` pushes, and the engagement PR, while fixture mode uses current-branch commits with recorded SHAs and no issue/comment/label/push/PR. Exercising the real GitHub trail live remains the recorded first-real-engagement residual risk.

## Validation Commands

The primary runner is `bash specs/c-suite-cpo-department/validate.sh` (authored by task `validate-all`): it implements exactly the per-AC checks below as functions with a per-AC pass/fail summary, exits non-zero on any failure, adds the cross-field equality assertions (probe prefix ↔ recorded prefix ↔ live `slash_commands`) and the AC12 negative test. The commands below are the script's normative contents and remain individually runnable.

- `for f in cpo-intake cpo-prd cpo-brief; do p=".claude/commands/c-suite/$f.md"; head -8 "$p" | grep -q "description:" || echo "FAIL $f description"; head -8 "$p" | grep -q "argument-hint:" || echo "FAIL $f argument-hint"; grep -q "^# " "$p" || echo "FAIL $f purpose"; for s in "## Variables" "## Instructions" "## Workflow"; do grep -q "^$s" "$p" || echo "FAIL $f $s"; done; grep -Eq "cpo-(question-bank|prd-standard|design-standard)" "$p" || echo "FAIL $f skill-ref"; done; echo checked` — verifies AC1.
