# Acceptance Criteria: c-suite CPO department

> The definition of "done" for [spec.md](./spec.md). Every criterion is observable and testable, and
> every task in tasks.md should map to at least one criterion here.

## Acceptance Criteria

- **AC1** — The three stage commands exist at `.claude/commands/c-suite/cpo-intake.md`, `cpo-prd.md`, `cpo-brief.md`; each has frontmatter with `description:` and `argument-hint:`, carries the house sections (a `#` Purpose heading, `## Variables`, `## Instructions`, `## Workflow`), and names at least one `cpo-*` knowledge skill it loads.
- **AC2** — The commands resolve under ONE canonical prefix proven by the naming probe: a headless listing shows all three under either `/c-suite:cpo-*` or `/cpo-*` (never a mix), and `.claude/rules/c-suite/cpo-operations.md` records that same canonical prefix. The three knowledge skills never auto-fire (frontmatter blocks model auto-invocation; the recorded legacy fallback applied and noted if needed).
- **AC3** — `cpo-intake.md` defines both modes (LIVE grilling; ASYNC question-list generation with send-to-client instructions) and re-entrancy: existing engagement detected and resumed, returned answers ingested into `discovery/client-answers.md`, and downstream stages marked `stale` when late answers contradict locked requirements.
- **AC4** — `cpo-prd.md` runs exactly two automatic Codex rounds, reads the verdict only from the report file (silence ≠ approval), upserts the round digest as an idempotent engagement-issue comment keyed `<!-- prd-review-round-N -->`, and carries the over-cap AskUserQuestion gate whose only options are one user-authorized final delta round (round 3 is last), accept-with-noted-gaps, or needs-human.
- **AC5** — All nine engagement templates exist inside their owning skills: `cpo-question-bank/templates/{question-list,requirements,status}.md`, `cpo-prd-standard/templates/prd.md`, `cpo-design-standard/templates/{project-brief,sitemap-and-flows,section-briefs,copy-deck,brand-inputs}.md`; the status template carries per-stage state + run-log fields and engagement-level issue/branch/PR fields.
- **AC6** — The three rules files exist under `.claude/rules/c-suite/` (`roster.md`, `cpo-operations.md`, `cpo-lessons.md`), each with `paths:` frontmatter (path-scoped, not always-loaded); `roster.md` classifies every seat as human or AI, includes the human prototype-owner seat, and stamps a model alias on every AI seat.
- **AC7** — The three knowledge skills pass the meta-skills validator and their frontmatter blocks auto-invocation.
- **AC8** — The five persona agents under `.claude/agents/cpo/` pass the meta-agent validator; none lists `AskUserQuestion` in `tools`; each declares its knowledge skill(s) in `skills:` frontmatter; and their `model` aliases match the roster (`opus` for cpo-pm and cpo-design-lead, `sonnet` for the rest).
- **AC9** — `.agents/skills/prd-review/SKILL.md` mirrors the spec-review contract: first-line verdict `### Round N — Verdict: approved|changes-requested` (em-dash), single report file `products/<client-slug>/prd/reviews/codex-prd-review-round-N.md`, terse two-line CLI return, closing `**Issue-comment digest:**`, judges against `requirements.md` + the cpo-prd-standard structure, and never calls `gh`.
- **AC10** — `AGENTS.md` documents `products/` in Project Structure and points to the `.claude/rules/c-suite/` family and the CPO pipeline.
- **AC11** — The dry-run fixture `products/_example-bluebird-bakery/` is complete: `discovery/requirements.md` with every dimension locked, `prd.md` plus an `approved` verdict report under `prd/reviews/`, all six design files present by name (`project-brief.md`, `sitemap-and-flows.md`, `section-briefs.md`, `copy-deck.md`, `brand-inputs.md`, ≥1 page under `design/wireframes/`), `status.md` showing `intake`, `prd`, and `brief` all `done` plus the engagement `handed-off`, and no wireframe referencing an external URL.
- **AC12** — `cpo-intake.md` states the canonical slug rule `^[a-z0-9][a-z0-9-]{0,38}[a-z0-9]$` with the `_example-` fixture exception, validates it at preflight BEFORE any filesystem/branch/worktree/`gh` side effect, and STOPs on violation.
- **AC13** — The fixture's `status.md` proves the engagement GitHub trail: it records the engagement issue number, the engagement branch name, the PR URL, and a per-stage run log including each stage's lessons-check result ("none" allowed).

## Validation Commands

Run these to prove the criteria above. Map each command to the criteria it verifies.

- `for f in cpo-intake cpo-prd cpo-brief; do p=".claude/commands/c-suite/$f.md"; head -8 "$p" | grep -q "description:" || echo "FAIL $f description"; head -8 "$p" | grep -q "argument-hint:" || echo "FAIL $f argument-hint"; grep -q "^# " "$p" || echo "FAIL $f purpose"; for s in "## Variables" "## Instructions" "## Workflow"; do grep -q "^$s" "$p" || echo "FAIL $f $s"; done; grep -Eq "cpo-(question-bank|prd-standard|design-standard)" "$p" || echo "FAIL $f skill-ref"; done; echo checked` — verifies AC1. A pass prints only `checked`.
- `claude -p 'Reply with only the names of available skills or commands that contain "cpo-", one per line.' | grep -Ec "^(c-suite:)?cpo-(intake|prd|brief)$"` — verifies AC2 (resolution; must print `3`, all rows sharing one prefix form) together with `grep -Eiq "canonical.*prefix.*/(c-suite:)?cpo-" .claude/rules/c-suite/cpo-operations.md && echo PASS` (recorded prefix matches).
- `grep -Eq "autoInvoke: *false|disable-model-invocation: *true" .claude/skills/cpo-question-bank/SKILL.md .claude/skills/cpo-prd-standard/SKILL.md .claude/skills/cpo-design-standard/SKILL.md && echo PASS` — verifies AC2/AC7 (no auto-fire).
- `for w in ASYNC LIVE client-answers.md stale resume send; do grep -qi "$w" .claude/commands/c-suite/cpo-intake.md || echo "FAIL intake $w"; done; echo checked` — verifies AC3. A pass prints only `checked`.
- `for w in "prd-review-round" "report file" "needs-human" "accept" "round 3" "two automatic"; do grep -qi "$w" .claude/commands/c-suite/cpo-prd.md || echo "FAIL prd $w"; done; echo checked` — verifies AC4. A pass prints only `checked`.
- `for t in cpo-question-bank/templates/question-list.md cpo-question-bank/templates/requirements.md cpo-question-bank/templates/status.md cpo-prd-standard/templates/prd.md cpo-design-standard/templates/project-brief.md cpo-design-standard/templates/sitemap-and-flows.md cpo-design-standard/templates/section-briefs.md cpo-design-standard/templates/copy-deck.md cpo-design-standard/templates/brand-inputs.md; do test -f ".claude/skills/$t" || echo "FAIL $t"; done; for w in Issue Branch PR lessons; do grep -qi "$w" .claude/skills/cpo-question-bank/templates/status.md || echo "FAIL status $w"; done; echo checked` — verifies AC5. A pass prints only `checked`.
- `for r in roster cpo-operations cpo-lessons; do head -5 ".claude/rules/c-suite/$r.md" | grep -q "paths:" || echo "FAIL $r paths"; done; grep -qi "human" .claude/rules/c-suite/roster.md && grep -qi "prototype" .claude/rules/c-suite/roster.md && grep -Eq "opus|sonnet" .claude/rules/c-suite/roster.md || echo "FAIL roster content"; echo checked` — verifies AC6. A pass prints only `checked`.
