# Acceptance Criteria: c-suite CPO department

> The definition of "done" for [spec.md](./spec.md). Every criterion is observable and testable, and
> every task in tasks.md should map to at least one criterion here.

## Acceptance Criteria

- **AC1** — The three stage commands exist at `.claude/commands/c-suite/cpo-intake.md`, `cpo-prd.md`, `cpo-brief.md`; each has frontmatter with `description:` and `argument-hint:`, carries the house sections (a `#` Purpose heading, `## Variables`, `## Instructions`, `## Workflow`), and names at least one `cpo-*` knowledge skill it loads.
- **AC2** — The commands resolve under ONE canonical prefix proven on the machine-readable discovery surface — the `slash_commands` field of the headless init event, never a model reply: exactly three entries match `(c-suite:)?cpo-(intake|prd|brief)`, all under the same prefix form, and that form equals the `Canonical command prefix:` line recorded in `.claude/rules/c-suite/cpo-operations.md`. The three knowledge skills never auto-fire (frontmatter blocks model auto-invocation; the recorded legacy fallback applied and noted if needed).
- **AC3** — `cpo-intake.md` defines both modes (LIVE grilling; ASYNC question-list generation with send-to-client instructions) and re-entrancy: existing engagement detected and resumed, returned answers ingested into `discovery/client-answers.md`, and downstream stages marked `stale` when late answers contradict locked requirements.
- **AC4** — `cpo-prd.md` runs exactly two automatic Codex rounds, reads the verdict only from the report file (silence ≠ approval), upserts the round digest as an idempotent engagement-issue comment keyed `<!-- prd-review-round-N -->`, and carries the over-cap AskUserQuestion gate whose only options are one user-authorized final delta round (round 3 is last), accept-with-noted-gaps, or needs-human.
- **AC5** — All nine engagement templates exist inside their owning skills: `cpo-question-bank/templates/{question-list,requirements,status}.md`, `cpo-prd-standard/templates/prd.md`, `cpo-design-standard/templates/{project-brief,sitemap-and-flows,section-briefs,copy-deck,brand-inputs}.md`; the status template carries exact-format per-stage state lines (`<stage>: <state>`), run-log fields, and engagement-level `mode:`/issue/branch/PR fields.
- **AC6** — The three rules files exist under `.claude/rules/c-suite/` (`roster.md`, `cpo-operations.md`, `cpo-lessons.md`), each with `paths:` frontmatter (path-scoped, not always-loaded); `roster.md` classifies every seat as human or AI, includes the human prototype-owner seat, and stamps a model alias on every AI seat.
- **AC7** — The three knowledge skills pass the meta-skills validator and their frontmatter blocks auto-invocation.
- **AC8** — The five persona agents under `.claude/agents/cpo/` pass the meta-agent validator; none lists `AskUserQuestion` in `tools`; each declares its knowledge skill(s) in `skills:` frontmatter; their `model` aliases match the roster per agent (cpo-pm `opus`, cpo-design-lead `opus`, cpo-ux-researcher `sonnet`, cpo-ux-designer `sonnet`, cpo-ui-designer `sonnet`); the three stage commands' delegation briefs carry the owning `SKILL.md` + template paths (the preload-independent channel); and the smoke check covered whichever invocation-control field shipped — including, under the legacy fallback (which blocks `skills:` preloading), a spot-check that a deployed persona Reads the brief-carried `SKILL.md`.
- **AC9** — `.agents/skills/prd-review/SKILL.md` mirrors the spec-review contract: first-line verdict `### Round N — Verdict: approved|changes-requested` (em-dash), single report file `products/<client-slug>/prd/reviews/codex-prd-review-round-N.md`, terse two-line CLI return, closing `**Issue-comment digest:**`, judges against `requirements.md` + the cpo-prd-standard structure, and never calls `gh`.
- **AC10** — `AGENTS.md` documents `products/` in Project Structure and points to the `.claude/rules/c-suite/` family and the CPO pipeline.
- **AC11** — The dry-run fixture `products/_example-bluebird-bakery/` is complete: `discovery/requirements.md` carries a `locked` marker for each of the twelve question-bank dimensions, `prd.md` plus an `approved` verdict report under `prd/reviews/`, all six design files present by name (`project-brief.md`, `sitemap-and-flows.md`, `section-briefs.md`, `copy-deck.md`, `brand-inputs.md`, ≥1 page under `design/wireframes/`), `status.md` carries the exact state lines `intake: done`, `prd: done`, `brief: done` and `engagement: handed-off`, and no wireframe references an external URL.
- **AC12** — `cpo-intake.md` states the canonical slug rule `^[a-z0-9][a-z0-9-]{0,38}[a-z0-9]$` with the `_example-` fixture exception and validates it at preflight BEFORE any filesystem/branch/worktree/`gh` side effect; the validation script's negative test invokes intake headlessly with a malformed slug and asserts a STOP with zero `products/` side effects.
- **AC13** — The fixture's `status.md` proves the fixture-mode trail: `mode: fixture`, the hosting branch name, per-stage commit SHAs that exist in the hosting branch's history, and a per-stage run log including each stage's lessons-check result ("none" allowed). (The real-engagement issue/branch/PR fields are proven structurally in the status template under AC5; exercising them live is the recorded first-real-engagement residual risk.)

## Validation Commands

The primary runner is `bash specs/c-suite-cpo-department/validate.sh` (authored by task `validate-all`): it implements exactly the per-AC checks below as functions with a per-AC pass/fail summary, exits non-zero on any failure, adds the cross-field equality assertions (probe prefix ↔ recorded prefix ↔ live `slash_commands`) and the AC12 negative test. The commands below are the script's normative contents and remain individually runnable.

- `for f in cpo-intake cpo-prd cpo-brief; do p=".claude/commands/c-suite/$f.md"; head -8 "$p" | grep -q "description:" || echo "FAIL $f description"; head -8 "$p" | grep -q "argument-hint:" || echo "FAIL $f argument-hint"; grep -q "^# " "$p" || echo "FAIL $f purpose"; for s in "## Variables" "## Instructions" "## Workflow"; do grep -q "^$s" "$p" || echo "FAIL $f $s"; done; grep -Eq "cpo-(question-bank|prd-standard|design-standard)" "$p" || echo "FAIL $f skill-ref"; done; echo checked` — verifies AC1. A pass prints only `checked`.
- `names=$(claude -p 'ok' --output-format stream-json --max-turns 1 | jq -r 'select(.type=="system" and .subtype=="init") | .slash_commands[]' | grep -E "^(c-suite:)?cpo-(intake|prd|brief)$"); [ "$(echo "$names" | wc -l)" -eq 3 ] || echo "FAIL count"; [ "$(echo "$names" | sed -E 's/cpo-(intake|prd|brief)$//' | sort -u | wc -l)" -eq 1 ] || echo "FAIL mixed prefixes"; rec=$(grep -oE "Canonical command prefix: */(c-suite:)?cpo-" .claude/rules/c-suite/cpo-operations.md | grep -oE "/(c-suite:)?cpo-"); echo "$names" | head -1 | grep -q "^${rec#/}" || echo "FAIL recorded-prefix mismatch"; echo checked` — verifies AC2 (machine-readable resolution + single prefix + recorded-prefix equality). A pass prints only `checked`.
- `grep -Eq "autoInvoke: *false|disable-model-invocation: *true" .claude/skills/cpo-question-bank/SKILL.md .claude/skills/cpo-prd-standard/SKILL.md .claude/skills/cpo-design-standard/SKILL.md && echo PASS` — verifies AC2/AC7 (no auto-fire).
- `for w in ASYNC LIVE client-answers.md stale resume send; do grep -qi "$w" .claude/commands/c-suite/cpo-intake.md || echo "FAIL intake $w"; done; echo checked` — verifies AC3. A pass prints only `checked`.
- `for w in "prd-review-round" "report file" "needs-human" "accept" "round 3" "two automatic"; do grep -qi "$w" .claude/commands/c-suite/cpo-prd.md || echo "FAIL prd $w"; done; echo checked` — verifies AC4. A pass prints only `checked`.
- `for t in cpo-question-bank/templates/question-list.md cpo-question-bank/templates/requirements.md cpo-question-bank/templates/status.md cpo-prd-standard/templates/prd.md cpo-design-standard/templates/project-brief.md cpo-design-standard/templates/sitemap-and-flows.md cpo-design-standard/templates/section-briefs.md cpo-design-standard/templates/copy-deck.md cpo-design-standard/templates/brand-inputs.md; do test -f ".claude/skills/$t" || echo "FAIL $t"; done; for w in Issue Branch PR lessons; do grep -qi "$w" .claude/skills/cpo-question-bank/templates/status.md || echo "FAIL status $w"; done; echo checked` — verifies AC5. A pass prints only `checked`.
- `for r in roster cpo-operations cpo-lessons; do head -5 ".claude/rules/c-suite/$r.md" | grep -q "paths:" || echo "FAIL $r paths"; done; grep -qi "human" .claude/rules/c-suite/roster.md && grep -qi "prototype" .claude/rules/c-suite/roster.md && grep -Eq "opus|sonnet" .claude/rules/c-suite/roster.md || echo "FAIL roster content"; echo checked` — verifies AC6. A pass prints only `checked`.
- `uv run .claude/skills/meta-skills/scripts/quick_validate.py .claude/skills/cpo-question-bank .claude/skills/cpo-prd-standard .claude/skills/cpo-design-standard` — verifies AC7 (builder confirms exact invocation against the script's usage).
- `uv run .claude/skills/meta-agent/scripts/validate_agent.py .claude/agents/cpo/*.md && ! grep -l "AskUserQuestion" .claude/agents/cpo/*.md && for a in .claude/agents/cpo/*.md; do grep -q "skills:" "$a" || echo "FAIL $a skills"; done; for m in "cpo-pm opus" "cpo-design-lead opus" "cpo-ux-researcher sonnet" "cpo-ux-designer sonnet" "cpo-ui-designer sonnet"; do set -- $m; grep -q "model: *$2" ".claude/agents/cpo/$1.md" || echo "FAIL $1 model"; done; for f in cpo-intake cpo-prd cpo-brief; do grep -q "SKILL.md" ".claude/commands/c-suite/$f.md" || echo "FAIL $f brief-paths"; done; echo checked` — verifies AC8 (validator, tools, skills preload, per-agent model mapping, brief-carried SKILL.md paths). A pass prints only `checked`; the legacy-fallback spot-check is the smoke-check task's recorded evidence.
- `head -1 .agents/skills/prd-review/SKILL.md >/dev/null && grep -q "Verdict: approved" .agents/skills/prd-review/SKILL.md && grep -q "codex-prd-review-round" .agents/skills/prd-review/SKILL.md && grep -q "Issue-comment digest" .agents/skills/prd-review/SKILL.md && grep -qi "never call" .agents/skills/prd-review/SKILL.md && echo PASS` — verifies AC9.

- `grep -q '\^\[a-z0-9\]\[a-z0-9-\]{0,38}\[a-z0-9\]\$' .claude/commands/c-suite/cpo-intake.md && grep -q "_example-" .claude/commands/c-suite/cpo-intake.md && echo PASS` — verifies AC12 (structural); the behavioral half runs in validate.sh: invoke intake headlessly with slug `bad_slug!!`, assert the reply contains a STOP and `products/` gained no new entry (`ls products/ | diff` against a pre-snapshot).
