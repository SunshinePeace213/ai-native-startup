The verdict up front

Commands and skills are the same mechanism now — the official docs merged them, and a .claude/commands/ file supports the identical frontmatter (hooks, model, args, invocation control). The choice is packaging, not capability. For the five rungs, the repo's own decision tree says flat commands: they're user-invoked, sequential, and side-effecting, their templates are runtime client scaffolds (they belong in projects/_template/, not inside a skill folder), and the locked /soriza-design:* namespace is itself a commands-subfolder feature.

Self-maintenance isn't a skill feature either — it's the repo's memory-series + development-log contract, which the locked decisions already reuse. The cards below carry the evidence; dispose of each to shape the plan prompt.

How the skill machinery works

The mental model behind commands/skills in Claude Code, the vocabulary to prompt with, and the quality bar the repo's own meta-skills standard holds authors to.

Mental model — one mechanism, three loading levels

Level 1 — the listing (always loaded)

Every skill's description + when_to_use sits in the session's skill listing from the first turn, capped at 1,536 characters. This is the trigger surface — it's how Claude decides to reach for a skill, and it's a small recurring context cost paid by every session in the repo.

Level 2 — the body (loaded on invocation, then it stays)

When you type /name (or Claude auto-triggers it), the markdown body loads into context — and persists across turns for the rest of the session. That's why the house style says keep bodies terse: every line is a recurring token cost after the first load.

Level 3 — supporting files (read only when needed)

A skill directory can carry references/, scripts/, assets/ — the body points at them and Claude reads or executes them on demand. This is the one real structural feature a flat command file lacks, and it's the whole "progressive disclosure" story.

Vocabulary ladder

TermPlain definitionMechanismHow you'd use it here
SKILL.md The entrypoint file of a skill directory. .claude/skills/<name>/SKILL.md → /name. A flat .claude/commands/<name>.md produces the same command with the same frontmatter support. The rungs stay flat command files; only meta-*-style tooling with bundled material earns a directory.
description / when_to_use The trigger surface Claude reads to decide when a skill applies. Always in the listing; truncated at 1,536 chars combined. Claude undertriggers, so descriptions are written "a little pushy." Rungs don't need discoverability — they're invoked by name in a fixed order — so their descriptions can be short and literal.
disable-model-invocation "Only the human runs this." Frontmatter flag; blocks auto-triggering and preloading. Every pipeline command in this repo sets it. Set on all five rungs so Claude never fires :brief on its own because intake.md "looks ready."
user-invocable: false "Only Claude loads this — it's knowledge, not an action." Hides the skill from the / menu; Claude pulls it in when relevant. The shape a doctrine file could take — but this repo puts ambient doctrine in path-scoped rules instead (see card 5).
command-scoped hooks A hook that exists only while one command runs. hooks: block in the command's frontmatter — the spec-gate pattern harness-plan.md already uses for its Stop hook. The intake DoR gate: a Stop hook in :intake's frontmatter, not a global entry in settings.json.
string substitutions Dynamic values expanded into the body before Claude sees it. $1/$ARGUMENTS for args; ${CLAUDE_PROJECT_DIR} for repo-root paths; ${CLAUDE_SKILL_DIR} for skill-bundled files. /soriza-design:intake <client> → $1 names the client folder; hook paths use ${CLAUDE_PROJECT_DIR}.
context: fork Run the skill in its own subagent instead of the main conversation. Frontmatter field (+ optional agent: type); keeps heavy work out of the session's context. Not needed for slice 1 — but it's the cheap upgrade path if a rung (e.g. wireframes) ever gets too heavy for one session.
paths: Glob patterns limiting when a skill auto-activates. Same format as path-specific rules; the skill loads only when work touches matching files. Rules and skills share this scoping idea — .claude/rules/soriza-design/ scoped to projects/**/* is the rules-side twin.

The quality bar (per this repo's meta-skills standard)

  • Bodies follow the five-section house template — Purpose · Variables · Instructions · Workflow · Report — in terse, imperative prose.
  • SKILL.md stays under 500 lines; detail moves to supporting files the body points at.
  • Trigger accuracy is tested with evals (with-skill vs baseline subagents), not assumed.
  • Never ship a same-name skill and command — the skill silently wins; delete the loser.
  • State what to do, never why — rationale is context bloat the agent never acts on.
