### Round 4 — Verdict: changes-requested

- **The complete-client marker sweep breaks the session-scoped gate it is meant to support.** The intake command's first write creates its own marker, but decisions.md, spec.md's DoR edge case, task 4, and AC7 then require sweeping every session's marker from already-complete clients. On a re-run of a complete client this can delete the current session's new marker; under same-client concurrency it can delete another live session's marker. That session's Stop hook then follows the separate “no own-session marker → exit 2” rule and is stranded. Fix: remove the cross-session/complete-client sweep from decisions.md, spec.md, task 4, and AC7–AC8; let successful Stop handling remove only its own markers, and add regressions for re-running a complete client and for one live session not deleting another's marker.
- **Several validation commands still accept builds that violate their criteria.** AC3's script accepts any five fetched `design/` entries without checking the five required source identities, their URLs/files, or their index entries; AC6's pipe count accepts empty columns or even one row containing several staff names; AC7 never checks the `Reads:`, `DoR gate:`, `Refusal:`, or `Commit:` field semantics; and AC11 does not prove there are exactly two gate-point bullets or reject additional/contradictory gate mappings. Fix: strengthen the assertion scripts in acceptance-criteria.md to compare AC3 against the exact required source/topic set and index, parse AC6 into exactly one four-nonempty-field row per staffer, assert AC7's required field clauses, and compare AC11's normalized gate-point set exactly to the two locked mappings.

**Issue-comment digest:** Round 4, changes-requested — 2 blocking: the complete-client marker sweep can delete live session markers and strand intake runs, and AC3/AC6/AC7/AC11 validation still permits contract-violating builds. Next: remove cross-session sweeping, add marker regressions, strengthen the four assertion groups, then re-review.
