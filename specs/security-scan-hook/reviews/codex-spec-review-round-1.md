### Round 1 — Verdict: changes-requested

- **The Bash tracker cannot provide the promised coverage for files that were already dirty at SessionStart.** `track_bash_writes.py` is specified to add only `current dirty set − baseline`, so an agent can modify a baseline-dirty file through Bash and that path remains excluded even though the Objective says every agent-written file is scanned and the selected design promises full Bash coverage. Fix: reconcile `spec.md` Objective/Non-Goals and `decisions.md`, then change the tracking design and Tasks 3–4 to detect changes caused by each Bash call (for example, a pre/post Bash snapshot) or explicitly narrow the guarantee and acceptance criteria to exclude this case.
- **The Stop-loop policy contradicts the cached hook guidance and cannot sustain the stated hard completion gate.** The plan requires continuing to block when `stop_hook_active` is true and treats the eight-block override as a backstop, but `ai-docs/anthropic/hooks-guide.md` (“Stop hook hits the block cap”) says the hook must parse `stop_hook_active` and exit early when true; it also states that Claude Code overrides a Stop hook after eight blocks without progress. As written, a secret can therefore survive the turn after the runtime override, contradicting the Objective. Fix: revise `spec.md`, the locked Stop-loop decision, Task 4, and AC5 to a documented, achievable guarantee; handle `stop_hook_active` as documented and move any required hard enforcement to an earlier event (including immediate scanning of Bash-created changes), or explicitly state the Stop gate's runtime limit.
- **AC8's registration validation does not verify what it claims.** `jq -r '.hooks | keys[]' .claude/settings.json` prints only event names; it cannot show or assert the two PostToolUse matchers, the five handler entries, or their script commands, despite claiming the Bash matcher is visible. Fix: replace or supplement the AC8 command with `jq -e` assertions over the matcher and command values for all five registrations (and retain focused checks for the gitignore and documentation entries).

**Recommendations (advisory, non-blocking):**

- Keep one Stop registration only if the same handler is genuinely needed for both parent and subagent completion; otherwise the immediate per-write and Bash scans may make the second sweep redundant and simplify state ownership.

**Issue-comment digest:** Round 1, changes-requested — 3 blocking: baseline subtraction misses Bash edits to already-dirty files, the Stop-loop policy contradicts documented behavior and the hard-gate guarantee, and AC8 does not verify the five hook registrations. Next: revise the coverage and Stop designs, strengthen AC8 validation, then re-review.
