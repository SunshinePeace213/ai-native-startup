### Round 1 — Verdict: changes-requested

- **The pre-worktree Artifact flow contradicts the grounded publishing contract.** tasks.md Steps 2/AC1–AC2 require medium/complex HTML to be drafted in an undefined “session scratchpad,” published before the worktree exists, and copied into the project later. `ai-docs/anthropic/artifacts.md` states that Claude writes the page to an HTML or Markdown file **in the project, then publishes it** (Create an artifact, line 49); the plan neither defines a project-local source file at publish time nor changes the workflow that enters the worktree only after grilling. Fix: revise decisions.md, tasks.md Step 2, and AC1–AC2 so artifact-producing grilling occurs after entering a worktree and publishes the canonical project-local file, or document a cached official contract that supports publishing the specified scratchpad path.
- **The ship-brief commit is sequenced after the commit it must join.** tasks.md Step 3 and AC5 put authoring in the Finish step before `gh pr ready`, but the current harness-build verification branches already create the approval-round report commit and designate it as the approved head before Finish runs. The brief therefore cannot be committed “together with” that already-created commit as written. Fix: update tasks.md Step 3 and AC5 to place brief authoring in every approval path immediately before the approval report commit (round 1, round 2, and approved round 3), leaving Finish to publish/update the PR body and verify that exact head.
- **Deviation-note ownership conflicts with the build lead’s no-edit contract.** tasks.md Step 3 says the lead creates and folds `implementation-notes.md` at each checkpoint, while `.claude/commands/harness-layer/harness-build.md` requires the lead to “never edit files directly” and route changes to subagents. An implementer must violate one instruction or invent an owner. Fix: assign creation and checkpoint folding to a named builder/recorder subagent in tasks.md and AC3, or explicitly scope a lead-owned exception in the planned harness-build instructions and acceptance criterion.
- **The allowed-path validation is guaranteed to fail on the plan branch.** tasks.md Step 6 requires `git diff origin/main --name-only` to contain no path outside spec.md’s Relevant Files list, but the branch already contains the four plan files and `specs/unknowns-aware-pipeline/artifacts/proposal.html`, none of which is in that implementation allowlist; this review report will also appear in the diff. Fix: validate implementation paths against the pre-build plan SHA, or explicitly allow the plan folder’s baseline files, generated review reports, implementation notes, and artifacts.
- **Most prompt-contract validation commands do not verify the criteria they claim to prove.** The AC1/AC2 command only finds three tokens; the AC3–AC5 command only finds `implementation-notes`, `ship brief`, and `deviation`; AC6, AC9, and AC10 similarly pass on a single keyword while omitting required ordering, gates, ownership, exclusions, dispositions, PR-body behavior, stale-brief handling, and the ≤2-line constraint. Fix: replace these keyword smoke checks in acceptance-criteria.md with targeted tests or explicit structural assertions for every required clause, and map each AC to the checks that actually prove it.

**Issue-comment digest:** Round 1, changes-requested — 5 blocking: an unsupported pre-worktree Artifact flow, impossible ship-brief/report commit ordering, conflicting deviation-note ownership, a guaranteed-failing changed-path allowlist, and validation commands that do not prove their criteria.
